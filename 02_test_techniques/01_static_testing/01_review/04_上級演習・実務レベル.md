# 🎯 レビュー技法 上級演習・実務レベル

## 🚀 実務レベル演習

### 🏢 演習1: 大規模プロジェクトのレビュー戦略

#### 📊 プロジェクト背景
```
プロジェクト: 金融機関向け基幹システム刷新
規模: 開発者30名、期間18ヶ月
成果物: 要件定義書200ページ、設計書500ページ、コード100万行
品質要件: 障害発生率 < 0.001%
```

#### 🎯 課題設定
**あなたの役割**: 品質保証責任者
**ミッション**: 効果的なレビュー戦略の立案・実行

#### 📋 検討事項

**1. レビュー対象の優先順位付け**

リスク影響度マトリックス：
| 成果物 | 変更頻度 | 影響範囲 | 複雑度 | 優先度 |
|--------|----------|----------|--------|--------|
| アーキテクチャ設計書 | 低 | 大 | 高 | ? |
| API仕様書 | 中 | 大 | 中 | ? |
| 業務ロジック詳細設計 | 高 | 中 | 高 | ? |
| DB設計書 | 低 | 大 | 中 | ? |
| 画面設計書 | 高 | 小 | 低 | ? |

**演習課題1**: 上記5つの成果物の優先度を決定し、レビュー技法を割り当ててください。

<details>
<summary>解答例・解説</summary>

**優先度付け（高→低）**:
1. **アーキテクチャ設計書** - 最優先（影響範囲大・複雑度高）
2. **DB設計書** - 高優先（影響範囲大・修正困難）
3. **業務ロジック詳細設計** - 高優先（変更頻度高・複雑度高）
4. **API仕様書** - 中優先（影響範囲大・複雑度中）
5. **画面設計書** - 低優先（影響範囲小・複雑度低）

**技法割り当て**:
- アーキテクチャ設計書: **インスペクション**（最重要・徹底検証）
- DB設計書: **インスペクション**（構造的影響大）
- 業務ロジック詳細設計: **ウォークスルー**（変更多・議論必要）
- API仕様書: **ピアレビュー＋チェックリスト**（効率重視）
- 画面設計書: **ピアレビュー**（軽量・迅速）
</details>

**2. レビュー体制の設計**

**演習課題2**: 30名体制での効率的なレビュー体制を設計してください。
- レビューチーム構成
- 役割分担
- スケジュール調整

<details>
<summary>解答例・解説</summary>

**レビュー体制案**:

**🎯 横断レビューチーム（6名）**:
- アーキテクト×2名
- 上級開発者×2名  
- テストリーダー×1名
- 品質保証×1名

**📋 領域別レビューチーム（各4-5名）**:
- フロントエンド：UI/UX専門
- バックエンド：API/DB専門
- インフラ：性能/セキュリティ専門

**🔄 ローテーション制**:
- 週次でチーム間メンバー交換
- 知識共有と偏りの防止

**⏰ スケジュール**:
- 重要文書：2週間サイクル
- 通常文書：1週間サイクル
- 緊急対応：48時間レスポンス
</details>

### 🔍 演習2: 複雑なレビューシナリオ

#### 📋 シナリオ設定
```
状況: 仕様変更により既存設計書の大幅修正が必要
制約: 
- 修正期間：2週間
- 影響範囲：5つのモジュール
- レビューリソース：限定的（3名のみ利用可能）
- 品質要件：妥協不可
```

#### 🎯 課題解決

**演習課題3**: 限られたリソースで効果的なレビューを実施する戦略を立案してください。

<details>
<summary>解答例・解説</summary>

**効率化戦略**:

**1. 差分レビュー重視**:
- 変更点の明確化（変更管理表作成）
- 影響分析マトリックス作成
- 変更部分に集中したレビュー

**2. ハイブリッドアプローチ**:
- 重要変更：インスペクション（1名×2日）
- 中程度変更：ウォークスルー（2名×半日）
- 軽微変更：ピアレビュー（1名×1時間）

**3. 並行レビュー**:
```
第1週：
- 月-火：モジュールA,B（並行）
- 水-木：モジュールC,D（並行）
- 金：全体整合性確認

第2週：
- 月-火：モジュールE + 統合確認
- 水-木：修正確認・フォローアップ
- 金：最終承認
```

**4. チェックリスト活用**:
- 変更パターン別チェックリスト
- 自動確認可能項目の事前処理
- レビュー時間の短縮
</details>

### 💼 演習3: レビュー会議の実践運営

#### 🎯 会議シナリオ
```
レビュー会議: API設計書インスペクション
参加者: 6名（アーキテクト、開発者×3、テスター、QA）
時間: 90分
レビュー対象: REST API仕様（20エンドポイント）
事前準備: 各自2時間で個人レビュー実施済み
```

#### 📋 実践課題

**演習課題4**: 効果的な会議進行シナリオを作成してください。

<details>
<summary>解答例・解説</summary>

**90分会議進行案**:

**🎯 開始（10分）**:
- 目的・ゴール確認
- 参加者の準備状況確認
- 進行ルール説明（建設的議論、時間管理）

**🔍 系統的レビュー（60分）**:
```
エンドポイント単位（3分/件）:
1. 機能概要確認（30秒）
2. 指摘事項報告（90秒）
3. 議論・合意（60秒）

優先度付け：
- Critical：即座に修正必要
- Major：リリース前修正
- Minor：改善提案として記録
```

**📝 まとめ（20分）**:
- 全指摘事項の分類・優先度確認
- 修正担当者・期限の決定
- フォローアップ方法の合意
- 次回レビューの必要性判断

**進行のコツ**:
- タイムボックス厳守
- 感情論回避、事実ベース議論
- 全員参加の促進（沈黙者への働きかけ）
</details>

### 📊 演習4: レビュー効果の定量分析

#### 📈 メトリクス分析課題

**与えられたデータ**:
```
プロジェクトA（レビュー実施）:
- レビュー工数：200時間
- レビューで検出された欠陥：85件
- テスト工程で検出された欠陥：15件
- 本番で発見された欠陥：2件

プロジェクトB（レビュー未実施）:
- テスト工程で検出された欠陥：120件
- 本番で発見された欠陥：18件
```

**演習課題5**: レビューのROIと効果を定量的に分析してください。

<details>
<summary>解答例・解説</summary>

**定量分析結果**:

**1. 欠陥検出効率**:
```
プロジェクトA:
- 総欠陥数：102件（85+15+2）
- レビュー検出率：83.3%（85/102）
- 本番流出率：2.0%（2/102）

プロジェクトB:
- 総欠陥数：138件（120+18）
- 本番流出率：13.0%（18/138）
```

**2. コスト効果分析**:
```
修正コスト想定：
- レビュー段階：1時間/件
- テスト段階：3時間/件
- 本番段階：20時間/件

プロジェクトA修正コスト：
- レビュー：85×1 = 85時間
- テスト：15×3 = 45時間
- 本番：2×20 = 40時間
- 合計：170時間

プロジェクトB修正コスト：
- テスト：120×3 = 360時間
- 本番：18×20 = 360時間
- 合計：720時間

ROI = (720-170-200)/200 = 175%
```

**3. 品質向上効果**:
- 本番欠陥85%削減（18件→2件）
- 顧客影響の大幅軽減
- ブランド価値の保護
</details>
