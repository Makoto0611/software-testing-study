# 🎯 同値分割法 上級演習・実務レベル

## 🎯 上級演習の位置づけ

### 📊 実務レベルスキルの定義
**到達目標**: 複雑な実務仕様に対して、効率的で実践的な同値分割を独立して実行できる

#### 🔥 上級スキルの特徴
- **複雑条件の整理**: 多重条件・例外処理を含む仕様への対応
- **効率性重視**: テストコストと品質のバランス最適化
- **実務適応**: プロジェクト制約・チーム状況を考慮した設計
- **継続改善**: 適用結果から手法をカスタマイズ・改善

### 🎯 基本演習との違い

| 観点 | 基本演習 | 上級演習 |
|------|----------|----------|
| **仕様複雑度** | 単純・明確 | 複雑・曖昧・例外多数 |
| **条件数** | 1-3個 | 5-10個以上 |
| **制約事項** | 技術的制約のみ | ビジネス・プロジェクト制約 |
| **期待成果** | 正確な適用 | 効率的・実践的な設計 |
| **評価基準** | 完全性 | 実務価値・ROI |

## 🛠️ 上級演習1: 金融商品申込システム

### 📋 詳細仕様

#### 🏦 システム概要
**機能**: オンライン投資信託申込受付システム
**対象商品**: 
- 国内株式型（手数料1.2%）
- 海外株式型（手数料1.8%）
- 債券型（手数料0.8%）
- バランス型（手数料1.5%）

#### 📊 申込条件
**顧客属性条件**:
- **年齢**: 20歳以上75歳以下
- **投資経験**: なし/1年未満/1-3年/3-5年/5年以上
- **年収**: 300万円以上（学生・主婦は除く）
- **投資目的**: 元本保証志向/安定志向/成長志向/積極志向

**申込金額条件**:
- **最小申込額**: 商品により異なる（1万円/5万円/10万円）
- **投資上限**: 年収の30%まで（投資経験による緩和あり）
- **商品別上限**: 
  - 海外株式型: 投資額の50%まで
  - その他: 制限なし

#### 🎯 承認ルール
**自動承認条件** (すべて満たす場合):
- 投資経験3年以上
- 申込額100万円以下
- 成長志向または積極志向
- 海外株式型の比率30%以下

**要審査条件** (いずれか該当):
- 投資経験1年未満
- 申込額500万円超
- 元本保証志向で株式型選択

**承認拒否条件**:
- 年収の30%超申込
- 75歳超または20歳未満
- 反社チェック該当

### 🔍 演習課題

#### 📝 課題1: 条件整理・優先順位付け（30分）
複雑な仕様を整理し、テスト設計の方針を決定してください。

**作成物**:
1. **条件マップ**: 全条件の関係図
2. **優先順位**: テスト重要度順の条件リスト
3. **分割方針**: 効率的な同値分割アプローチ

#### 📝 課題2: 同値クラス設計（60分）
実務的な観点で同値クラスを設計してください。

**考慮事項**:
- ビジネスリスクの高い条件を重視
- テスト実行コストを考慮
- 実際のユーザー行動パターンを反映

**作成物**:
1. **同値クラス一覧**: 分割根拠付き
2. **代表値設定**: 実行可能な具体値
3. **リスク評価**: 各クラスのビジネスリスク

#### 📝 課題3: テストケース最適化（45分）
プロジェクト制約を考慮したテストケース設計

**制約条件**:
- テスト実行期間: 3日間
- テスト担当者: 2名
- 自動化不可（手動テストのみ）

**最適化要件**:
- 重要機能の十分な網羅
- 実行効率の最大化
- 欠陥検出確率の最大化

### 💡 解答アプローチ例

#### 🎯 条件整理のポイント
**主要分類軸**:
1. **顧客セグメント**: 経験・年収・年齢による分類
2. **商品特性**: リスク・手数料・最小金額
3. **承認フロー**: 自動承認・要審査・拒否

**優先順位付け基準**:
- **Prio1**: 承認可否に直結（年収・年齢・反社）
- **Prio2**: ビジネス影響大（申込額・商品選択）
- **Prio3**: ユーザビリティ（画面表示・メッセージ）

#### 🎯 効率的分割戦略
**複合条件の扱い**:
- 独立条件は個別に分割
- 相互依存条件は組み合わせパターンで分割
- 稀なケースは代表ケースに集約

**実務的代表値選択**:
- 典型的な顧客プロファイル（30代会社員、40代主婦など）
- 実際の商品価格帯
- 一般的な投資金額

## 🛠️ 上級演習2: ECサイト複合検索機能

### 📋 詳細仕様

#### 🛒 システム概要
**機能**: 大手ECサイトの商品検索・絞り込み機能
**データ規模**: 100万件以上の商品、1000以上のカテゴリ

#### 🔍 検索条件
**基本検索**:
- **キーワード**: 部分一致、AND/OR検索、除外キーワード
- **カテゴリ**: 大分類（20種）→中分類（200種）→小分類（1000種）
- **価格帯**: 下限・上限指定、定型範囲選択

**詳細絞り込み**:
- **ブランド**: 複数選択可、人気順・五十音順
- **評価**: 4.5以上/4.0以上/3.5以上/指定なし
- **送料**: 無料のみ/指定なし
- **在庫**: 在庫ありのみ/予約可含む/すべて
- **発送**: 当日発送/翌日配送/指定なし

#### 📊 表示・ソート
**ソート条件**:
- 関連度順（デフォルト）
- 価格安い順/高い順
- 新着順
- レビュー数順
- 評価高い順

**表示件数**: 20件/50件/100件
**ページング**: 最大500ページまで

#### 🎯 パフォーマンス要件
- **応答時間**: 3秒以内（90%のケース）
- **同時接続**: 1000ユーザーまで
- **検索精度**: 関連商品上位20件に80%以上

### 🔍 演習課題

#### 📝 課題1: テスト戦略設計（45分）
複雑な検索システムに対する効果的なテスト戦略を設計

**考慮点**:
- 機能面・性能面・使いやすさの バランス
- データ準備の制約
- 自動化可能部分の見極め

#### 📝 課題2: 同値分割の応用設計（90分）
従来の同値分割を拡張・応用した設計手法を検討

**チャレンジ**:
- 組み合わせ爆発への対応
- 現実的なユーザー行動の反映
- エッジケース・異常系の効率的な網羅

#### 📝 課題3: テストデータ設計（60分）
テスト実行に必要なデータ設計と準備方針

**要件**:
- 代表的なユーザージャーニーをカバー
- パフォーマンステストも考慮
- 継続的なテスト実行が可能

## 🛠️ 上級演習3: 医療システム薬剤処方チェック

### 📋 詳細仕様

#### 🏥 システム概要
**機能**: 電子カルテシステムの薬剤処方安全性チェック
**重要性**: 医療安全に直結、エラーは重大事故につながる

#### 💊 チェック項目
**基本チェック**:
- **用法・用量**: 年齢・体重・疾患別基準値
- **併用禁忌**: 薬剤間相互作用データベース
- **アレルギー**: 患者の既知アレルギー歴
- **疾患禁忌**: 患者の既往歴・現在疾患

**詳細チェック**:
- **重複投与**: 同一成分・同効薬の重複
- **投与経路**: 経口・注射・外用の適応性
- **投与期間**: 長期投与時の安全性
- **腎・肝機能**: 患者の臓器機能による減量要否

#### ⚠️ アラートレベル
**Level 1 (情報提供)**: 一般的な注意事項
**Level 2 (注意)**: 用量調整推奨
**Level 3 (警告)**: 処方変更推奨
**Level 4 (禁止)**: 処方不可、理由必須

#### 🎯 特殊要件
- **緊急時オーバーライド**: 医師判断で警告無視可能
- **妊婦・小児**: 特別な安全基準適用
- **法的要求**: 医薬品副作用被害救済制度対応

### 🔍 演習課題

#### 📝 課題1: 安全性重視の同値分割（120分）
医療安全の観点を最優先した同値分割設計

**重点項目**:
- 重大事故につながるリスクシナリオの特定
- 法的・倫理的要求事項の反映
- 医療現場での実用性確保

#### 📝 課題2: 境界条件の特別扱い（60分）
医療システム特有の境界条件への対応

**特殊境界**:
- 小児用量（体重1kg当たりの計算）
- 高齢者減量（年齢・腎機能による調整）
- 緊急時例外処理

#### 📝 課題3: 継続的品質保証（45分）
運用開始後の継続的なテスト・品質確保の仕組み

**考慮点**:
- 薬事データベースの更新対応
- 新薬追加時のテスト方針
- 実運用での異常検知・改善
